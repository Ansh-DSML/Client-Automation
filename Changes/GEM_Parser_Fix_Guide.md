# GEM Parser Fix Guide
## Based on Analysis of 3 Real GeM PDFs

---

## Part 1 — What the PDFs Actually Contain (Ground Truth)

All three PDFs are generated by `wkhtmltopdf` from the GeM portal. They are
**plain text-layer PDFs** (not scanned). The Hindi font (`Mangal-Regular`) uses
CID encoding, which causes `(cid:N)` garbage in pdfplumber's table cells for
Hindi text. The English content extracts cleanly from the same table cells.

There are **NO embedded Excel/CSV attachments** (`pdfdetach` returns 0 files).
There are **NO annotation URLs** (`BoqLineItemsDocument` links do not exist in
these PDFs). The only data source is the PDF text itself.

## Part 3 — Key Observations & Patterns

### O1: Part Number IS the numeric prefix of Item Category
Every schedule's `Item/Category` text begins with a numeric HAL part number followed by either a space or a hyphen:
```
33505964011, 80 PIN MODULE FLANGE PLUG       → part = 33505964011
51107159 Crystal Oscillator 36.864 Mhz...   → part = 51107159
51126961-circular Connector, C Size...       → part = 51126961
68838104 Copper Braid Hal-6176...            → part = 68838104
```
**Extraction regex:** `r'^(\d[\d-]*)\s*[-,\s]'` — grab the leading digit sequence up to the first non-digit separator.

### O2: Description IS the full Item/Category text from the schedule table
The full text in the `Item/Category` column of the **Evaluation Schedules table** is exactly the right content for the `Description` column. It contains the HAL part number + item name. No truncation is needed — use the full string.

### O3: The Evaluation Schedules table has TWO layouts
- **3-column:** `Schedule | Item/Category | Quantity` (PDFs 1 and 3)
- **4-column:** `Schedule | Estimated Value | Item/Category | Quantity` (PDF 2)

The parser must detect column count and select the right column indices.

### O4: Delivery Days differ per schedule and must be matched per row
Each schedule has its own consignee block appearing after it. The consignee tables appear in order (S1 consignees, then S2 consignees, etc.). Delivery Days come from the last column of each consignee table row. They can differ per schedule (e.g., PDF 2: S1=179, S2=180, S3=180).

### O5: The Evaluation Schedules table is always on page 4 (0-indexed: page index 3)
All 3 PDFs place the schedule table at page 4. However, it's better to scan all pages and detect the table by its header keywords.

### O6: Consignee tables appear in the same order as schedules
There are always exactly N consignee tables (one per schedule). They are the 5-column tables with structure:
`S.No | Consignee | Address | Quantity | Delivery Days`
The `***` in the address column identifies them reliably.

### O7: Bid Number format is always `GEM/YYYY/B/NNNNNNN`
Reliable regex: `r'GEM/\d{4}/[A-Z]/\d+'`

### O8: Organisation Name always contains the short code in parentheses
```
Hindustan Aeronautics Limited (hal)   → short = HAL
```
Extract with: `r'Organisation Name\s+(.+?)(?:\n|$)'` then look for `(xxx)` at the end.

### O9: There are NO embedded files, NO BOQ CSV URLs in these PDFs
The existing `download_boq_csv_from_annotations()` and `parse_boq_excel()` functions
will always return empty for this PDF type. The data is entirely self-contained in the
PDF text.

### Change 2 — REPLACE `parse_schedules_from_tables()` with a robust version

The current function has two failure modes:
1. It requires `"schedule"` in `str(table[0]).lower()` — but in these real PDFs,
   `table[0]` is the Hindi header row full of `(cid:N)` garbage, so the check fails.
2. It looks for rows starting with `"Schedule N"` but only if the header check passes.

**Replace the entire function with:**

```python
def parse_schedules_from_tables(pdf_bytes: bytes) -> list[dict]:
    """
    Parse evaluation schedules from the PDF's Evaluation Schedules table.

    Table structure (detected by the data rows, NOT the garbled Hindi header):
      3-column: ['Schedule N', 'Item/Category text', 'Quantity']
      4-column: ['Schedule N', 'Estimated Value', 'Item/Category text', 'Quantity']

    Returns list of dicts: {schedule_no, part_no, description, qty}
    """
    items = []
    try:
        with pdfplumber.open(io.BytesIO(pdf_bytes)) as pdf:
            for page in pdf.pages:
                tables = page.extract_tables()
                for table in tables:
                    if not table or len(table) < 2:
                        continue

                    # Detect this is the Evaluation Schedules table by checking
                    # whether any data row starts with "Schedule N"
                    schedule_rows = []
                    for row in table:
                        if not row:
                            continue
                        cell0 = str(row[0] or "").strip()
                        m = re.match(r'Schedule\s+(\d+)', cell0, re.IGNORECASE)
                        if m:
                            schedule_rows.append((int(m.group(1)), row))

                    if not schedule_rows:
                        continue

                    # Determine layout: 3-col or 4-col
                    # Detect by checking if column 1 looks like a large number (Estimated Value)
                    # vs a descriptive string (Item/Category)
                    for sched_no, row in schedule_rows:
                        ncols = len(row)
                        if ncols >= 4:
                            # Check if col[1] is a numeric estimated value
                            col1 = str(row[1] or "").strip().replace(",", "")
                            if re.match(r'^\d+(\.\d+)?$', col1):
                                # 4-col layout: col[2]=description, col[3]=qty
                                description = str(row[2] or "").strip()
                                qty = str(row[3] or "").strip()
                            else:
                                # 4-col but col[1] is actually description
                                description = str(row[1] or "").strip()
                                qty = str(row[-1] or "").strip()
                        elif ncols == 3:
                            # 3-col layout: col[1]=description, col[2]=qty
                            description = str(row[1] or "").strip()
                            qty = str(row[2] or "").strip()
                        else:
                            continue

                        # Extract part number from the leading numeric token
                        part_no = _extract_part_no_from_category(description)

                        if description and qty:
                            items.append({
                                "schedule_no": sched_no,
                                "part_no":     part_no,
                                "description": description,
                                "qty":         qty,
                            })
    except Exception as e:
        log.warning(f"Table-based schedule extraction failed: {e}")
    return items
```

### Change 3 — ADD new helper `_extract_part_no_from_category()`

This is the single correct way to get the part number from these PDFs:

```python
def _extract_part_no_from_category(category_text: str) -> str:
    """
    Extract the HAL/GeM part number from an Item Category string.

    Patterns observed:
      '33505964011, 80 PIN MODULE FLANGE PLUG'  → 33505964011
      '51107159 Crystal Oscillator 36.864 Mhz'  → 51107159
      '51126961-Circular Connector, C Size...'  → 51126961
      '68838104 Copper Braid HAL-6176...'        → 68838104

    Strategy: grab the leading digit-only token (may include a trailing hyphen
    that separates the PN from the description).
    """
    # Strip leading/trailing whitespace
    text = category_text.strip()

    # Match a leading numeric block (digits only, no letters)
    m = re.match(r'^(\d+)', text)
    if m:
        return m.group(1).strip()

    # Fallback: return the whole string if no numeric prefix found
    return text.upper()
```

### Change 4 — REPLACE `parse_consignee_from_tables()` to return per-schedule delivery days

The current function returns a flat list that is then matched by index (fragile).
Replace it with a version that returns one delivery_days value per schedule:

```python
def parse_consignee_delivery_days(pdf_bytes: bytes) -> list[str]:
    """
    Extract delivery days from each consignee table, in schedule order.

    Consignee tables are 5-column tables:
      [S.No | Consignee | Address | Quantity | Delivery Days]
    They are identified by '***' in the Address column (col index 2).

    Returns a list of delivery_days strings, one per schedule, in order.
    e.g. ['179', '180', '180'] for a 3-schedule bid.
    """
    delivery_days = []
    try:
        with pdfplumber.open(io.BytesIO(pdf_bytes)) as pdf:
            for page in pdf.pages:
                tables = page.extract_tables()
                for table in tables:
                    if not table or len(table) < 2:
                        continue
                    for row in table:
                        if not row or len(row) < 5:
                            continue
                        # Address column (index 2) contains '***'
                        addr = str(row[2] or "")
                        if "***" not in addr:
                            continue
                        # Delivery days is the last column
                        days = str(row[-1] or "").strip()
                        if days.isdigit():
                            delivery_days.append(days)
    except Exception as e:
        log.warning(f"Consignee delivery days extraction failed: {e}")
    return delivery_days
```

### Change 9 — FIX `parse_text_schedules_fallback` regex for these PDFs

The existing `parse_schedules_from_text()` regex will fail on these PDFs because
the schedule table in plain text looks like:

```
 Schedule 1  Connector, 33505964011, 80 Pin Module Flange Plug  20
 Schedule 2  Connector, 63500004000, 80 Pin Connector Kit       20
```

The current regex expects a short alphanumeric part number between `Schedule N`
and the quantity. Here the "part number" is actually the full description string.

Update the fallback text regex to:

```python
def parse_schedules_from_text(text: str) -> list[dict]:
    """
    Text-based fallback schedule extraction.
    Looks for lines like: 'Schedule N  <description text>  <qty>'
    """
    items = []
    pattern = re.compile(
        r'Schedule\s+(\d+)\s+'       # Schedule N
        r'(.+?)\s+'                   # description (non-greedy)
        r'(\d+)\s*$',                 # trailing quantity
        re.IGNORECASE | re.MULTILINE
    )
    for m in pattern.finditer(text):
        sched_no  = int(m.group(1))
        desc_raw  = m.group(2).strip()
        qty       = m.group(3).strip()
        part_no   = _extract_part_no_from_category(desc_raw)
        items.append({
            "schedule_no": sched_no,
            "part_no":     part_no,
            "description": desc_raw,
            "qty":         qty,
        })
    return items
```

## Part 8 — Summary of All Bugs Fixed

| # | Bug | Root Cause | Fix |
|---|---|---|---|
| 1 | Wrong description (template string) | PATH C used `f"Refer specification for {part}"` | New `parse_schedules_from_tables()` returns real description from table |
| 2 | Wrong part number (demo data) | `_items_match_mpr_parts()` triggered on partial match | Remove hardcoded matching functions entirely |
| 3 | Wrong part number (full description used) | `parse_schedules_from_tables()` failed silently due to CID garbage in header check | New function detects table by data row pattern, not header content |
| 4 | Wrong/missing L/T | Single bid-level delivery days used for all rows | `parse_consignee_delivery_days()` returns per-schedule values |
| 5 | Description truncated | Schedule table `Item/Category` column was truncated in old regex | New function reads full table cell, no regex truncation |
| 6 | Part number not extracted correctly | Old code used `row[1]` directly as part_no | New `_extract_part_no_from_category()` strips numeric prefix |
| 7 | 4-column vs 3-column table not handled | Old code assumed fixed column indices | New code detects layout by checking if col[1] is numeric |
